## 1. 问题诊断：为什么计划了还是做不到

### 1.1 核心发现

用户的痛点**不是缺少任务管理工具**。用户已经有分类、优先级、任务列表——形式完整，但执行依然极度混乱。问题出在：**计划与执行之间缺少一个实时、主动的连接层。**

### 1.2 三层问题模型

#### 层级 A：启动冻结（Decision Paralysis）

| 症状 | 心理机制 | 现有产品缺失 |
|------|----------|-------------|
| 面对 20 条任务却无法动手 | Hick's Law：选项越多，决策时间越长 | 只展示列表，不替用户做选择 |
| 不知从何开始，面对电脑发呆 | 决策疲劳 + 任务模糊性 | 无「第一步」引导 |
| 从小被安排，缺乏自驱启动能力 | 外部执行系统依赖（父母/老师/老板） | 无主动推送机制 |

#### 层级 B：上下文蒸发（Context Evaporation）

| 症状 | 心理机制 | 现有产品缺失 |
|------|----------|-------------|
| 被打断后忘了在做什么 | 工作记忆只有 4±1 个槽位 | 无上下文保存/恢复 |
| 中途插入事情后偏离计划 | 注意力跟随外部刺激而非内部计划 | 无偏离检测与提醒 |
| 不知道自己是否应该被打断 | 无法实时判断打断的优先级 | 无打断决策辅助 |

#### 层级 C：拖延螺旋（Procrastination Spiral）

| 症状 | 心理机制 | 现有产品缺失 |
|------|----------|-------------|
| 忙了一天但什么都没完成 | 无目的切换 = 虚假忙碌 | 无「有效工作」检测 |
| 拖延 → 内疚 → 焦虑 → 更拖延 | 负反馈循环 + 情绪性回避 | 日终只展示负面数据 |
| 手机/社交媒体持续分散注意力 | 多巴胺劫持 + 注意力碎片化 | 干扰屏蔽力度不足 |
| 可能进一步导致抑郁 | 习得性无助（Seligman） | 无正反馈机制 |

### 1.3 核心洞察

> 用户描述的模式本质上是 **「被动响应式注意力」** ——注意力跟着外部刺激走，而非被内部计划驱动。
>
> 这不是个人缺陷，而是**一个从未被训练过的能力**。从小到大，父母、老师、老板扮演了「外部执行系统」的角色。当这个外部指令源消失，需要自驱时，大脑没有现成的回路可用。
>
> **Trace 需要扮演的角色：一个温和但坚定的「内化执行教练」**——不是监工，不是老板，而是一个始终在场、主动引导、在你脱轨时温和拉你回来的系统。最终目标：在使用过程中，逐渐帮用户建立起自己的执行功能。

## 2. 理论框架：行为心理学基础

产品中的每一个设计决策都基于经过验证的心理学原理。以下是核心理论支柱。

### 2.1 执行功能理论（Executive Function）

执行功能包括：**工作记忆**、**认知灵活性**、**抑制控制**。这三项能力决定了你能否「计划 → 启动 → 维持 → 完成」一件事。ADHD 研究表明，执行功能可以通过外部支架（scaffold）来增强。

**产品应用**：「此刻该做什么」功能充当外部工作记忆；「打断拦截」充当外部抑制控制。

### 2.2 时间感知障碍（Time Blindness）

拖延者往往有「时间盲」——无法准确感知时间的流逝。10 分钟和 60 分钟在主观上没有明显差异。这导致计划不切实际、截止日期总是突然到来。

**产品应用**：常驻状态条实时显示已用时间/剩余时间；时间块让时间「可见」。

### 2.3 习惯回路（Habit Loop — Duhigg）

提示(Cue) → 惯常行为(Routine) → 奖励(Reward)。拖延本身也是一个习惯回路：焦虑(提示) → 刷手机(惯常行为) → 暂时缓解(奖励)。要改变行为，需要**替换惯常行为并保留奖励**。

**产品应用**：检测到「漫游模式」时提供替代行为（帮你选一个任务开始），完成后给予正反馈（宠物奖励）。

### 2.4 自我决定理论（SDT — Deci & Ryan）

内在动机来自三个基本心理需求：**自主性(Autonomy)**、**胜任感(Competence)**、**归属感(Relatedness)**。监控式工具破坏自主性，过难的任务破坏胜任感。

**产品应用**：每个提醒都保留「别管我」选项（自主性）；「只要 15 分钟」降低门槛（胜任感）；宠物提供情感连接（归属感）。

### 2.5 实施意图（Implementation Intentions — Gollwitzer）

「我打算在 X 时间、X 地点做 X 事」的 if-then 计划比模糊的目标意图（「我要写报告」）执行率高 2-3 倍。关键是预先绑定时间、地点和行为。

**产品应用**：时间块系统将任务绑定到具体时段；晨间启动仪式帮用户形成 if-then 计划。

### 2.6 情绪调节与拖延（Sirois & Pychyl）

拖延的核心不是时间管理问题，而是**情绪管理问题**。人们回避任务是因为任务引发了不愉快的情绪（焦虑、无聊、挫败），拖延是短期情绪调节策略。

**产品应用**：「换一个任务」时收集原因数据；AI 识别情绪性回避模式并提供温和的认知重构建议。

### 2.7 Zeigarnik 效应

未完成的具体行为比未完成的抽象目标更能吸引大脑注意。一旦写下具体的「第一步」，大脑会持续惦记它。

**产品应用**：每个任务必填「第一步」字段——「打开文档，从第三节开始」而非「写报告」。

### 2.8 Foot-in-the-door 效应

一旦开始一个小行动，人会倾向于继续。15 分钟的承诺远比 2 小时容易接受，而实际上大多数人开始后会继续超过 15 分钟。

**产品应用**：启动助推的「只要 15 分钟就好」。

## 3. 设计原则

基于以上理论，产品的每一个干预都遵循三条核心原则：

| 原则 | 含义 | 反面教材 |
|------|------|---------|
| **① 降低启动摩擦** | 让开始一件事变得比不开始更容易 | 让用户面对 20 条列表自己选 |
| **② 增加脱轨摩擦** | 让偏离计划变成一个有意识的决定，而非无意识的滑动 | 用户切走时不做任何提醒 |
| **③ 建立正反馈闭环** | 让每一次「完成」都被看见、被庆祝、被记住 | 日终只展示「5 个任务未完成」 |

补充设计准则：

- **产品是教练，不是监工**：所有提醒语气温和，所有功能都可关闭，用户拥有最终控制权
- **脚手架式教育**：初期高度辅助，随用户能力增长逐步撤除干预，最终目标是用户不再需要系统的主动引导
- **数据驱动个性化**：基于用户历史行为数据，让建议越来越精准
- **先表扬后改进**：任何反馈场景都先展示成就，再温和提出改进空间

### 4.2 架构关键点

执行守护层**不需要重构现有架构**，而是在 Tauri 原生层和 React UI 之间加入一个新的事件处理管道：

1. **感知层**持续产出事件流（当前活跃窗口、空闲状态、切换频率）
2. **计划层**持续提供期望状态（当前时间块应该做什么）
3. **执行守护层**实时比对两者，在偏差超出阈值时触发干预

技术实现路径：在 Zustand store 中新增 `executionGuardian` slice，监听 activity 和 task 数据变化，驱动 UI 层的提醒弹窗和状态更新。

## 6. 逐页详解：每一页的功能与设计理据

所有页面按用户使用流程排列。每个功能说明「是什么」「为什么这样设计」「在心理训练中扮演的角色」。

---

### 6.1 Dashboard 仪表盘

**定位**：主页，用户每天的第一个接触点
**版本**：V1
**对应文件**：`src/pages/Dashboard.tsx`, `src/components/Dashboard/`

#### 功能 A：晨间启动仪式 `[V1] [新增]`

**是什么**：打开 app 时不是空白桌面，而是一个结构化的启动流程：
1. 显示今日时间块概览
2. 展示昨日未完成任务
3. 确认/调整计划
4. 一键进入第一个任务

**为什么**：实施意图理论(Gollwitzer)证明，预先确认「几点做什么」比模糊计划执行率高 2-3 倍。晨间仪式将 if-then 计划仪式化。

**心理角色**：实施意图 · 降低启动摩擦

**实现要点**：
- 新组件 `src/components/Dashboard/MorningRitual.tsx`
- 检测当日是否首次打开（store 记录 `lastMorningRitualDate`）
- 从 Planner 数据读取今日时间块，从任务列表读取昨日未完成项
- 底部 CTA：「开始第一个任务」直接联动此刻引擎
- 可在 Settings 中关闭

#### 功能 B：此刻引擎 · 单一任务展示 `[V1] [新增]`

**是什么**：Dashboard 中央不再是 todo list，而是一个大字展示：
- 「现在 → 写季度报告」+ 预估时间 + 已拖延天数
- 只给一个任务，只需点「开始」
- 点「换一个」需说明原因（V2 增加原因收集）

**为什么**：Hick's Law——选项越多决策时间越长。当任务列表有 20 条时，大脑在选择上花的精力可能比做任务本身还多。给出一个答案，而不是一个列表。

### 6.3 Focus 专注模式

**定位**：沉浸式执行 · 打断防护 · 启动助推
**版本**：V1
**对应文件**：`src/pages/FocusMode.tsx`, `src/components/FocusStartedModal.tsx`, `src/components/FocusCompletedModal.tsx`

#### 功能 A：启动助推 · 第一步引导 `[V1] [新增]`

**是什么**：点击「开始专注」后，不是直接开始计时，而是先弹出一个轻量引导：
1. 「你打算先做哪一小步？」
2. 给出 AI 建议的 2-3 个具体动作 + 自定义输入框
3. 底部提示：「只要 15 分钟就好。」
4. 确认后才开始倒计时

**为什么**：启动是拖延的最大障碍。"只要15分钟"利用了 foot-in-the-door 效应——一旦开始，惯性会让你继续。具体的第一步消除了模糊性（模糊是拖延的温床）。

**心理角色**：Foot-in-the-door · 去模糊化

**实现要点**：
- 修改 `FocusStartedModal.tsx`，在现有启动流程中增加一步
- 显示任务的 `first_step` 字段内容，并提供 AI 生成的额外建议
- 新增 15 分钟快捷计时选项（与现有番茄钟 25 分钟并列）
- 用户确认的第一步保存到 focus session 数据中

#### 功能 B：打断拦截器 `[V1] [新增]`

**是什么**：专注执行期间，如果检测到切换到计划外应用超过 2 分钟，弹出温和提醒：
- 「你已经离开 [当前任务] X 分钟了。」
- 三个选项：
  1. **回去继续** — 直接恢复
  2. **只是看一眼**（3min 倒计时）— 限时偏离
  3. **确实有急事，暂停任务** — 触发上下文快照

**为什么**：很多时候人不是故意拖延，是真的没意识到自己已经切走了。一个温和的提醒就足以重新激活抑制控制（执行功能的一部分）。三个选项尊重自主性（SDT），避免变成监控工具。

**心理角色**：抑制控制激活 · 自主性保护

**实现要点**：
- 新组件 `src/components/InterruptionAlert.tsx`
- 监听感知层的窗口切换事件，在 Focus session 期间判断是否偏离
- 偏离阈值可在 Settings 中配置（默认 2 分钟，宽松 5 分钟，严格 1 分钟）
- 选择「只是看一眼」时启动 3 分钟倒计时浮窗
- 所有打断事件记录到数据库，供统计使用

#### 功能 C：上下文快照 `[V1] [新增]`

**是什么**：选择「暂停任务」时自动保存：
- 当前任务名 + 已投入时间
- 用户写的一句话备注（弹窗快速输入"做到哪了"）
- 恢复时显示：「你之前在做 X，已投入 Y 分钟，你的备注是 Z，要继续吗？」

**为什么**：工作记忆只有 4±1 个槽位。一次打断足以清空全部上下文。快照充当外部工作记忆，消除"回到之前状态"的认知成本。

**心理角色**：外部工作记忆 · 恢复成本归零

**实现要点**：
- 数据模型新增 `context_snapshots` 表：`{task_id, elapsed_time, note, created_at, resumed_at}`
- 新组件 `src/components/ContextSnapshot.tsx`（保存弹窗 + 恢复卡片）
- 暂停时触发保存弹窗（简单文本输入 + 一键保存）
- Dashboard 和 StatusBar 显示未恢复的快照卡片

#### 功能 D：已有功能保留 `[V1]`

番茄钟计时器（25/5/15 可配）、呼吸动画、环境音（雨声/咖啡馆/森林等）、干扰网站屏蔽、宠物迷你组件、专注会话记录与 XP 奖励。这些已有功能与新增的执行守护功能形成完整的专注闭环。

**定位**：全新页面 · 系统托盘/桌面悬浮窗 · 全局可见

**是什么**：检测模式：
- 5 分钟内切换窗口超过 15 次、每个停留不到 20 秒 → 判定为「漫游」
- 或者：在非生产力应用上停留超过阈值、无明确任务关联

**为什么**：很多人的拖延表现不是"什么都不做"，而是"什么都做一点但都不深入"。漫游检测识别这种无意识的注意力分散，在用户陷入空转时提供一个决策锚点。

## 7. 用户引导：产品内教育与能力培养

### 7.1 设计哲学：脚手架式教育

「脚手架」(Scaffolding) 是教育学概念：提供临时支撑，让学习者完成原本无法独立完成的任务，然后逐步撤除支撑。Trace 的执行守护系统就是脚手架——初期高度辅助，随着用户能力增长，逐渐降低干预频率，最终用户能独立管理执行。

### 7.2 知识卡片系统（Knowledge Cards）`[V2]`

在产品的关键触点嵌入 30 秒可读完的心理学知识卡片。不是独立的「学习中心」，而是在上下文中自然出现。

| 触发场景 | 卡片内容 |
|----------|---------|
| 第一次被打断拦截时 | 「**你知道吗？** 研究表明，被打断后平均需要 23 分钟才能完全恢复到之前的专注状态。这就是为什么我们在你切走时提醒你——不是不信任你，而是帮你省下那 23 分钟。」 |
| 第一次使用「只要15分钟」 | 「**为什么是15分钟？** 心理学中的"foot-in-the-door"效应：一旦开始一个小行动，你的大脑会自然想要继续。大多数人在开始 15 分钟后会选择继续——因为启动才是最难的部分。」 |
| 日终复盘时看到时间黑洞 | 「**关于时间盲**：很多人低估了"刷一会儿"的实际时间。这不是自控力差，而是大脑天生不擅长感知时间流逝。这就是为什么我们让时间始终可见——帮你建立准确的时间感知。」 |
| 连续 3 天回避同一个任务 | 「**拖延不是时间管理问题**，而是情绪管理问题。你回避这个任务，可能是因为它引发了不愉快的情绪（焦虑？无聊？害怕失败？）。试试把它拆成一个 10 分钟的小步骤，降低情绪门槛。」 |
| 第一次晨间仪式 | 「**为什么要确认计划？** 研究表明，"我打算在X时间做X事"的具体计划比"我今天要努力"的模糊意图，执行率高出 2-3 倍。你现在做的就是在形成这种高效计划。」 |
| 首次日终复盘 | 「**为什么先看成就？** 自我批评不会减少拖延，反而会加重。自我同情（承认进展，温和面对不足）才是打破拖延循环的关键。所以我们先看你做了什么，再看改进空间。」 |

**实现要点**：
- 新组件 `src/components/KnowledgeCard.tsx`
- 卡片数据存储在 `src/data/knowledgeCards.ts`（JSON 格式，方便后续国际化）
- 每张卡片有唯一 ID，用户看过后记录到 store（不重复展示）
- 卡片可收藏，在设置或独立页面中回顾

### 7.3 能力成长可视化 `[V3]`

追踪并展示用户在以下维度的成长趋势，让能力增长变得可见：

| 能力维度 | 计算方式 | 数据来源 |
|----------|---------|---------|
| 时间感知准确度 | 预估时间 vs 实际时间的偏差率趋势 | 任务预估 + 实际追踪 |
| 打断恢复速度 | 从脱轨到恢复执行的平均时间趋势 | 打断事件记录 |
| 自主启动率 | 不需要系统提醒就主动开始的比例 | 此刻引擎使用 vs 主动开始 |

### 7.4 渐进式脚手架撤除 `[V3]`

系统根据用户行为数据自动调整干预强度：

| 阶段 | 条件 | 干预水平 |
|------|------|---------|
| **初期** | 默认 | 高频提醒。晨间仪式自动弹出，打断检测灵敏，每 25 分钟状态确认，日终复盘自动触发。像一个认真的新手教练。 |
| **中期** | 连续 2 周打断恢复速度 < 5min 且自主启动率 > 60% | 降低干预频率。打断检测延迟到 3 分钟，状态确认改为 45 分钟。像一个放手的教练。 |
| **后期** | 各项指标持续达标 1 个月 | 进入「安静模式」——只在检测到明显异常时才介入。用户已经内化了执行能力。像一个在场但不干预的顾问。 |

### 7.5 Onboarding 引导流程优化 `[V1]`

首次使用时的引导从「功能教学」升级为「理念引导」：不只是教你怎么用按钮，而是告诉你为什么这样设计——

> 「我们不会给你一个 20 条的任务列表让你选，因为选择本身就消耗精力。我们每次只告诉你一件事。」

让用户从一开始就理解产品的设计理念，建立信任。

**实现要点**：
- 修改 `src/components/Onboarding.tsx`，增加执行守护相关引导步骤
- 引导步骤：此刻引擎 → 状态常驻条 → 打断拦截 → 晨间/日终 → 设置个性化参数

在 Focus session 期间，监听窗口切换事件：

```typescript
class DeviationDetector {
  private currentTask: Task;
  private allowedApps: string[]; // 任务关联的应用
  private sensitivity: 'relaxed' | 'standard' | 'strict';

  // 灵敏度对应的阈值（秒）
  private thresholds = { relaxed: 300, standard: 120, strict: 60 };

  onWindowChange(newApp: string, newTitle: string) {
    if (this.isAllowedApp(newApp)) return;

    // 开始计时
    this.deviationTimer = setTimeout(() => {
      this.triggerInterruptionAlert();
    }, this.thresholds[this.sensitivity] * 1000);
  }

  onReturnToAllowedApp() {
    clearTimeout(this.deviationTimer);
  }
}
```

- [ ] 超过阈值时触发提醒

**实现描述**：

检测逻辑：
```typescript
class WanderingDetector {
  private switchBuffer: number[] = []; // 最近 5 分钟的切换时间戳
  private thresholds = {
    relaxed: { switches: 20, window: 300 },
    standard: { switches: 15, window: 300 },
    strict: { switches: 10, window: 300 },
  };

  onWindowSwitch() {
    this.switchBuffer.push(Date.now());
    this.cleanOldSwitches();

    if (this.switchBuffer.length >= this.currentThreshold.switches) {
      this.triggerWanderingAlert();
    }
  }
}
```

### 11.2 状态管理约定

所有执行守护相关状态统一放在 `useAppStore` 的 `executionGuardian` slice 中。避免创建独立 store。

```typescript
// 读取状态
const recommendedTask = useAppStore(s => s.executionGuardian.currentRecommendedTask);

// 触发 action
const { refreshRecommendedTask } = useAppStore(s => s.executionGuardianActions);
```

### 11.3 事件流架构

```
┌──────────────┐
│  Tauri 层     │  每秒产出 window_change 事件
│  (Rust)       │  通过 tauri event 发送到前端
└──────┬───────┘
       │ tauri://event
       ↓
┌──────────────┐
│ trackingService│  接收事件，更新 activity store
│  (已有)       │
└──────┬───────┘
       │ store 订阅
       ↓
┌──────────────────────────────────┐
│ deviationDetector / wanderingDetector │
│  (新增 services)                      │
│  监听 store 变化，判断偏离/漫游        │
└──────┬───────────────────────────┘
       │ 触发 store action
       ↓
┌──────────────┐
│  UI 层        │  InterruptionAlert / WanderingAlert
│  (React)      │  弹窗组件响应 store 状态变化
└──────────────┘
```

### 11.4 关键注意事项

1. **不破坏现有功能**：所有新增功能通过 feature flag 控制，可单独开关
2. **性能**：偏离检测和漫游检测在前端 service 层运行，不在渲染循环中；使用 `requestIdleCallback` 或 debounce
3. **隐私**：打断事件数据存储在本地，不上传云端（除非用户主动开启同步）
4. **国际化**：所有新增文案同步更新 `src/i18n/zh-CN.json` 和 `src/i18n/en-US.json`
5. **无障碍**：弹窗支持键盘导航和 ESC 关闭
6. **测试**：每个新组件需有对应的 Playwright 测试用例

### 11.5 后端 API 扩展

如需云端同步（V2+），后端需新增以下 REST 端点：

```
POST   /api/v1/guardian/snapshots          # 创建快照
GET    /api/v1/guardian/snapshots           # 获取活跃快照
PATCH  /api/v1/guardian/snapshots/:id       # 恢复快照
POST   /api/v1/guardian/interruptions       # 记录打断
GET    /api/v1/guardian/interruptions/stats  # 打断统计
POST   /api/v1/guardian/reviews             # 保存日终复盘
GET    /api/v1/guardian/reviews/:date       # 获取某日复盘
POST   /api/v1/guardian/wandering           # 记录漫游事件
GET    /api/v1/guardian/settings            # 获取配置
PUT    /api/v1/guardian/settings            # 更新配置
```

V1 阶段所有数据仅存本地（通过 Tauri IPC），后端 API 在 V2 同步功能中实现。
